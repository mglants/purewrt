# PureWRT

[English](README.md) · **Русский**

Нативный для OpenWrt менеджер маршрутизации для прокси [mihomo](https://github.com/MetaCubeX/mihomo).

> **OpenWrt решает, куда направить трафик. Mihomo занимается прокси.**

PureWRT оставляет решения о маршрутизации там, где им и место — в межсетевом экране: dnsmasq
перенаправляет DNS на резолвер mihomo, получает ответы с реальными IP и наполняет наборы (sets)
nftables; nftables сопоставляет IP назначения и через TPROXY заворачивает **только выбранный трафик** в
посекционные слушатели mihomo. Всё остальное идёт напрямую (или через mwan3). Провайдеры правил
разворачиваются в nftset/nftables, а не в движок `rules:` mihomo — поэтому «горячий путь» проходит через
ядро, а не через прокси.

---

## Как это работает

```
Клиент в LAN
   │  DNS-запрос
   ▼
dnsmasq ──перенаправляет──► резолвер mihomo (127.0.0.1:7874) ──► ответ с реальным IP
   │                                                               │
   │  наполняет наборы nftables (по секциям)  ◄────────────────────┘
   ▼
nftables: IP назначения есть в наборе секции?
   ├─ да  → TPROXY в слушатель mihomo этой секции → прокси / VPN / zapret
   └─ нет → напрямую (или mwan3 / маршрут по умолчанию)
```

Режим по умолчанию — **маршрутизация по реальным IP через nftset**. Режимы fake-ip и full-core
намеренно не используются по умолчанию. Несопоставленный трафик никогда не загоняется в прокси
принудительно.

### Цели маршрутизации — не только mihomo

Секция маршрутизации не привязана к прокси mihomo. Каждая секция отправляет сопоставленный трафик через
тот бэкенд, который вы выберете:

- **группа прокси mihomo** — обычный путь через прокси (selector / url-test / load-balance / …).
- **VPN** — направить трафик секции прямо в интерфейс WireGuard/VPN, настроенный на роутере, вообще без
  прокси.
- **zapret** — применить обход DPI к трафику секции без какого-либо прокси или VPN.
- **direct** / **reject** — обойти всё либо отбросить.

Так что можно свободно комбинировать: например, стриминг — через группу прокси, рабочую подсеть — в
туннель WireGuard, заблокированные сайты — через zapret, а всё остальное — напрямую. (Конфигурации VPN и
Zapret даже сохраняются при сбросе мастером, поскольку они ваши, а не импортированные.)

## Быстрый старт — мастер настройки

После установки пакета `purewrt` и LuCI-приложения `luci-app-purewrt`:

1. Откройте LuCI → **Services → PureWRT** (страница **General**).
2. Нажмите **Run Setup Wizard** (запустить мастер настройки).
   > ⚠️ Применение мастера **сбрасывает всю конфигурацию PureWRT к значениям по умолчанию** (подписки,
   > провайдеры, секции маршрутизации, привязки устройств, DNS, настройки). Сохраняются только ваши
   > конфигурации **VPN** и **Zapret**, бинарник mihomo и учётные данные контроллера.
3. **Шаг 1 — выберите источник:**
   - **Use a subscription URL** — вставьте URL подписки Clash/Mihomo, список прокси или список правил
     (рекомендуется, если провайдер дал вам URL).
   - **Default lists** — подтянуть готовые списки правил прямо из опубликованного каталога; при желании
     добавьте URL с узлами прокси.
   - **Manual setup** — пропустить импорт и настроить узлы/провайдеров позже из обычных вкладок.
4. **Предпросмотр** того, что будет импортировано, затем привяжите наборы правил к **секциям
   маршрутизации** и задайте протокол каждой секции (группа прокси / VPN / zapret / direct) на
   drag-and-drop доске.
5. Задайте глобальные параметры (IPv6, обновления, DNS) и **завершите** — мастер импортирует,
   генерирует и применяет всё за один шаг.

## Добавление подписки

Три способа — выберите подходящий:

- **Мастер** — Шаг 1 → *Use a subscription URL* (см. выше). Лучше всего для первоначальной настройки.
- **LuCI → вкладка Proxy Subscriptions** — добавьте подписку, вставьте URL (и при необходимости тип
  панели / заголовки), нажмите **Save & Apply**. Так можно добавить подписку, ничего не сбрасывая.
- **CLI:**
  ```sh
  purewrt analyze 'https://panel.example/sub/secret?format=clash'   # только предпросмотр (необязательно)
  purewrt import  'https://panel.example/sub/secret?format=clash'   # сохранить
  purewrt update                                                    # загрузить узлы + правила
  purewrt apply                                                     # генерация + применение
  ```

## Возможности

- **Мастер настройки** — сброс-и-старт или импорт; выбор URL подписки либо готовых списков по
  умолчанию, привязка наборов правил к секциям маршрутизации и настройка прокси/VPN/zapret по секциям —
  всё на одной drag-and-drop доске.
- **Подписки на прокси** — Mihomo/Clash YAML, списки proxy-URI, base64-подписки, с учётом особенностей
  панелей (Remnawave и совместимые) и стабильным HWID, выводимым из идентичности роутера.
- **Провайдеры правил** — текст, MRS (бинарный формат, потоковое декодирование без лишних аллокаций) и
  GeoSite/GeoIP из локальных dat-файлов v2ray. Импорт готовых списков прямо из опубликованного каталога
  с множественным выбором.
- **Секции / маршрутизация** — группировка правил в секции, каждая маршрутизируется через группу прокси,
  VPN, zapret либо direct/reject; стратегия прокси, фильтры и тип группы настраиваются по секциям.
- **Маршрутизация по устройствам** — привязка устройств LAN к секциям по MAC (переживает смену
  DHCP-аренды, охватывает IPv6).
- **Управление mihomo** — просмотр групп прокси, переключение узлов с тестом задержки и сливом
  соединений, отслеживание upstream-релизов alpha.
- **Интеграция zapret** — поставляется в комплекте, опциональна, определяется в рантайме и аккуратно
  отключается при отсутствии.
- **Сосуществование с mwan3** — режим «обнаружить и сосуществовать» для multi-WAN; маскированная
  fwmark сохраняет биты mwan3.
- **Диагностика** — проверка доменов, страница «Что сейчас заблокировано» и страница **Client Traffic**,
  которая в реальном времени снимает потоки + DNS + сигналы отклонений и помечает заблокированные,
  залипшие из-за DPI и «замороженные» соединения, обогащая их данными ASN/страны.
- **Эксплуатация** — экспорт/импорт конфигурации (секреты скрыты), push-уведомления (ntfy/webhook) о
  сбоях обновления и истечении подписок, метрики Prometheus с примерами дашбордов и алертов Grafana.

## Компоненты

| Бинарник / приложение | Назначение                                                          |
|-----------------------|---------------------------------------------------------------------|
| `purewrt`             | CLI-менеджер — импорт, обновление, генерация, применение, диагностика. |
| `purewrt-check`       | Классификатор доменов (в какую секцию/маршрут попадёт хост).         |
| `purewrt-api`         | Опциональный локальный API-демон.                                   |

`/etc/purewrt/generated/*`.

## Установка (репозиторий пакетов)

На роутере с OpenWrt **24.10** (opkg) или **25.12+** (apk) — одной строкой:

```sh
wget -O - https://mglants.github.io/purewrt/install.sh | sh
```

Скрипт сам определит релиз и архитектуру, добавит подписанный репозиторий PureWRT и его ключ доверия,
заменит `dnsmasq` на `dnsmasq-full` и установит `purewrt` + `luci-app-purewrt` (для опционального обхода
DPI через zapret добавьте `WITH_ZAPRET=1`). Затем откройте **LuCI → Services → PureWRT → Run Setup
Wizard**.

Хотите добавить репозиторий и установить вручную? `wget -O - https://mglants.github.io/purewrt/feed.sh | sh`,
затем `opkg install purewrt luci-app-purewrt` (или `apk add …`). Структура репозитория и публичные ключи —
по адресу `https://mglants.github.io/purewrt/`. Пакеты собираются в GitHub Actions для подобранного набора
архитектур (x86_64, aarch64 cortex-a53/a72/a76, mipsel_24kc); для прочих целей собирайте из исходников ниже.

## Установка и сборка

PureWRT собирается с помощью OpenWrt SDK. Входящий в комплект `Taskfile.yml` управляет загрузкой SDK и
сборками в dev-окружении NixOS для OpenWrt 24.10 и 25.12 и двух эталонных устройств.

```sh
# dev-окружение NixOS
nix-shell
task doctor            # проверить инструментарий
task build:25.12       # полная сборка SDK → артефакты .apk
task build:cudy-wr3000h
task build:bananapi-bpi-r3-mini
```

Либо собрать пакет в уже развёрнутом SDK:

```sh
./scripts/feeds update -a && ./scripts/feeds install -a
make package/mihomo-alpha/compile V=s
make package/purewrt/compile V=s
```

Зависимости в рантайме: `dnsmasq-full`, `nftables`, `kmod-nft-tproxy`, `ip-full`, `ca-bundle`,
`luci-base` и бинарник mihomo (входящий в комплект пакет `mihomo-alpha` предоставляет `/usr/bin/mihomo`).

> OpenWrt 25.12 использует пакеты APK, поэтому артефакты — это `.apk` (а не `.ipk`). См. заметки по
> сборке в конце файла про переопределение целей SDK и устранение проблем на NixOS.

## Конфигурация

Всё управляется через UCI. Например, подписка:

```uci
config subscription
    option url 'https://panel.example/sub/secret?format=clash'
    option panel_type 'remnawave'
    option user_agent 'PureWRT/0.2'
    list header 'X-Custom-Header: value'
```

PureWRT выводит стабильный HWID из идентичности роутера и подставляет `hwid`/`device_name`, если их нет;
при каждой загрузке он отправляет `x-hwid`, `x-ver-os`, `x-device-model` (плюс совместимые псевдонимы).
Ручное переопределение HWID намеренно игнорируется, чтобы идентичность оставалась выводимой из роутера.

## Маленькие роутеры / мало RAM

В PureWRT есть настройка `resource_profile` (`standard` (по умолчанию) / `low` / `high`). Задайте её во
вкладке **Settings** или командой `uci set purewrt.settings.resource_profile='low'`. Режим **`low`**
автоматически:

- переносит кэш правил в **tmpfs** (`/tmp/purewrt/cache`) вместо флеша;
- **отключает кэш артефактов** (провайдеры заново разбираются при каждом обновлении — экономит флеш и RAM);
- **выключает дедупликацию правил** (меньше RAM/CPU; дублирующиеся элементы наборов безвредны);
- **отключает маршрутизацию IPv6** при `ipv6_mode=auto` (переопределите `ipv6_mode=on`, если IPv6 реально нужен);
- **принудительно выключает geodata mihomo** и подстраивает ядро mihomo (без сопоставления процессов,
  короче keep-alive, проверки здоровья прокси выключены);
- уменьшает лимит резервной копии конфига при apply до 128 КБ.

Параметры, которые `low` **не** трогает, но их стоит отключить на слабом железе:

```uci
config main 'settings'
    option resource_profile 'low'
    option ipv6_mode 'off'
    option dashboard_enabled '0'          # дашборд metacubexd стоит ~5 МБ
    option mihomo_geodata_enabled '0'
    option background_updates '0'
    option update_concurrency '1'
```

Кроме того: **предпочитайте нативные списки из каталога** (`parse_mode=native_import`, путь мастера
«Default lists — самый лёгкий») и небольшие текстовые списки вместо больших провайдеров **MRS** или
**GeoSite/GeoIP** — декодирование MRS и посекционные буферы dnsmasq (ограничены 32 МБ) дают основные
всплески RAM. Подписки в режиме `low` и так импортируют только узлы прокси (а не свои большие списки
правил), если вы не включите это явно для конкретной подписки.

**Что потянет:**

| RAM | Что запускать |
|-----|---------------|
| **~32 МБ** | Ядро прокси mihomo (Go-бинарник, которому нужны десятки МБ) обычно **не помещается** рядом с OpenWrt. Используйте **маршрутизацию только через zapret** — направляйте секции в zapret для обхода DPI **без прокси** (см. [Цели маршрутизации](#цели-маршрутизации--не-только-mihomo)); сам слой маршрутизации/DNS лёгкий. Либо запускайте прокси на железе помощнее. |
| **~64 МБ** | Минимальный прокси mihomo возможен с `resource_profile=low`: несколько узлов, небольшие нативные списки, дашборд выключен, без geodata, IPv6 выключен. |
| **≥128 МБ** | Комфортно — профиль `standard`, провайдеры MRS, IPv6 и дашборд работают нормально. |

## Команды

```sh
purewrt analyze <url>        # изучить подписку без импорта
purewrt import <url>         # импортировать подписку / провайдера
purewrt add-native-list <url> <section> [--priority=N]
purewrt update               # загрузить подписки + провайдеры правил
purewrt generate             # сгенерировать артефакты в /etc/purewrt/generated
purewrt apply                # генерация + установка + перезагрузка сервисов (с самовосстановлением)
purewrt reload               # синоним apply
purewrt status               # текущее состояние
purewrt validate             # проверить конфигурацию
purewrt client-traffic <IP>  # живой захват потоков / DNS / отклонений для клиента
purewrt doctor               # проверить инструментарий разработки/рантайма
purewrt disable              # удалить изменения маршрутизации и DNS, внесённые PureWRT
purewrt-check chatgpt.com    # классифицировать домен
```

## Модель безопасности

- Внешний контроллер mihomo по умолчанию слушает на `127.0.0.1`.
- PureWRT владеет только `/etc/purewrt/generated/*`, своими фрагментами dnsmasq и `table inet purewrt`.
- Политика маршрутизации использует **маскированную** fwmark `0x1/0xff`; nftables ставит метку через
  `meta mark set meta mark | 0x1`, сохраняя биты mwan3.
- `apply` самовосстанавливается: перед тем как довериться кэшу отпечатка (fingerprint), он проверяет
  живое состояние (таблицу nft, ip rules, фрагменты dnsmasq), и одна сломанная подписка не прерывает всё
  обновление.
- `disable` удаляет только изменения маршрутизации и DNS, сделанные PureWRT.

## Разработка

```sh
task test:go     # go test ./... && go vet ./...  — канонический набор проверок
go test ./internal/manager/... -run TestApply
```

Версия Go-инструментария закреплена в `mise.toml`; `go.mod` объявляет минимальную версию языка. Внешние
зависимости сведены к минимуму: `gopkg.in/yaml.v3` и `github.com/klauspost/compress` (zstd, для
декодирования бинарного MRS).

Карту архитектуры и неочевидные подводные камни (сброс кэша LuCI, перезапуск-а-не-reload dnsmasq,
загрузочный `apply --force`, проверки самовосстановления), на каждый из которых есть регрессионные тесты,
см. в `CLAUDE.md` и `AGENTS.md`.

<details>
<summary>Цели OpenWrt SDK и устранение проблем на NixOS</summary>

### Цели SDK

По умолчанию используется `x86/64`. При необходимости переопределите переменные цели:

```sh
OPENWRT_VERSION=24.10.0 OPENWRT_TARGET=x86 OPENWRT_SUBTARGET=64 OPENWRT_ARCH=x86_64 task build
```

SDK загружается в `.build/openwrt/`; пакеты синхронизируются в SDK по пути
`package/network/services/`; итоговые файлы `.apk`/`.ipk` выводятся из `bin/packages` / `bin/targets`.

### Cudy WR3000H (mediatek/filogic, 25.12)

```sh
OPENWRT_VERSION=25.12.0 OPENWRT_GCC_VERSION=14.3.0 OPENWRT_TARGET=mediatek \
  OPENWRT_SUBTARGET=filogic OPENWRT_ARCH=x86_64 task build
```

### Выбор feed-ов

Taskfile намеренно **не** запускает `./scripts/feeds install -a` — на `mediatek/filogic` это подтягивает
посторонние варианты загрузчиков. Устанавливается только то, что нужно `purewrt` и `mihomo-alpha`. Если
SDK был «загрязнён» через `install -a`, запустите `task sdk:reset` с соответствующими переменными цели и
пересоберите.

### NixOS nix-ld

`shell.nix` экспортирует `NIX_LD`/`NIX_LD_LIBRARY_PATH` для совместимости со stub-ld. Если сборка упала на
`golang-bootstrap`, очистите артефакты host Go командой `task sdk:clean-go-host` (с переменными цели) и
пересоберите.

### Обновления mihomo

Перезапись `/usr/bin/mihomo` в рантайме отключена намеренно — PureWRT поставляет управляемый пакетом
`mihomo-alpha`. Обновляйте его пересборкой пакета (`openwrt/mihomo-alpha/Makefile`) или установкой более
новой версии пакета из вашего репозитория.

</details>
